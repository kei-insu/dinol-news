## 배포

### 배포 절차 (순서 고정 · 이 순서를 지킬 것)

1. **`git pull`** ← 제일 먼저, 파일을 배치·수정하기 **전에**. (로컬을 원격과 맞춘 clean 상태에서 시작)
   - `Already up to date.` 또는 조용한 업데이트면 정상. `CONFLICT`/`error:` 뜨면 **멈추고** 해결.
2. **파일 배치**: `Dinol_news_YYYYMMDD.html` → `news\YYYY\MM\`, `index.json` → 루트 덮어쓰기.
3. **대장 재생성**: `python scripts/build_published_urls.py` (→ "중복 등장 URL 없음" 확인).
4. **배포**: `./deploy.ps1 "add: M/D 브리핑"` (M/D는 실제 날짜로).

> ⚠️ **순서의 이유**: 파일을 먼저 얹고 pull하면(2→1 역순) 로컬·원격이 같은 파일을 고친 상태라 충돌난다. pull은 항상 clean 상태에서 맨 먼저. (2026-07-13 다운 사고가 이 역순 + 충돌 강행에서 발생, issues.md 참조)

## 톡톡 인증 · Firestore 보안

| 항목 | 가드레일 |
|---|---|
| 비밀번호 저장 | **평문 저장 금지.** SHA-256 해시만 저장·대조 |
| 클라이언트 검증의 한계 | 브라우저 JS의 검증은 **UX 편의일 뿐 보안 경계가 아니다.** 실제 방어는 **Firestore 보안 규칙**이 한다. 규칙 없이 클라이언트 체크만 믿지 말 것 |
| **운영자 권한** | ⛔ **코드에 마스터 비밀번호를 두지 않는다.** 운영자 판별은 **Firebase Auth 세션**으로만 한다(`dinolAdmin.login()`). 클라이언트 상수·해시 어느 쪽도 금지 — 공개 JS라 평문은 즉시, 해시는 브루트포스로 뚫린다 |
| 운영자 계정 | Firebase 콘솔 Authentication에 전용 계정 1개. **UID를 `firestore.rules`에 박아 권한 부여**. 폐기된 값(`481516`)은 이미 공개됐으므로 **어떤 비밀번호로도 재사용 금지** |
| Firestore 규칙 | git 아님 → **Firebase 콘솔에 직접 게시.** 글자수·구조 변경 시 규칙도 함께(4곳: 글 create/update, 댓글 create/update) |
| 규칙 범위 | `comments` 서브컬렉션 + `commentCount` 갱신 규칙 포함 |
| App Check | reCAPTCHA v3 **Enforce**(콘솔에서 강제). 규칙과 별개 레이어 |
